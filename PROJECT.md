`org-*.svg`는 조직 아바타를 base64로 내장한 SVG로, 조직당 1파일이다. 팔레트는 **테마 중립 단일 팔레트**(미디어 쿼리 없음)다. 글자는 라이트/다크 페이지 배경 양쪽에서 읽히는 중간톤 회색(#7d8590 계열), 조직 이름만 GitHub 블루(#4493f8)로 링크임을 표시하고, 카드 배경은 회색 7% 반투명 + 옅은 테두리다. 원래는 `@media (prefers-color-scheme: dark)` 2팔레트였으나 Safari가 `<img>` SVG의 미디어 쿼리를 지원하지 않아 iOS에서 카드마다 무작위 팔레트로 깨졌고(아래 절 함정 2), 2026-07 중립 팔레트로 전환했다. `<img>` 임베드에서는 SVG 내부 링크가 동작하지 않으므로, 카드를 조직당 1파일로 분리하고 README에서 각 `<img>`를 `<a href>`로 감싸 클릭 이동을 구현한다. 카드 하단에 투명 여백 8px를 넣어 세로 스택 시 카드 간격이 생기게 했다.

### 테마 대응 SVG의 동작 원리 (prefers-color-scheme)

**상태: 2026-07부터 org 카드는 이 방식을 쓰지 않는다.** 아래 함정 2(Safari 미지원, 비결정적 렌더링) 때문에 테마 중립 단일 팔레트로 전환했다. 이 절은 그 결정의 배경 지식이자, 같은 시도를 반복하지 않기 위한 기록으로 남긴다.

카드가 라이트/다크에서 다르게 보이는 것은 서버가 테마별로 다른 파일을 주기 때문이 아니다. **하나의 파일 안에 두 팔레트가 모두 들어 있고, 렌더링 시점에 브라우저가 한쪽을 골라 그리는 것**이다. 세 가지 요소의 조합으로 동작한다.

**1. SVG 내부 CSS는 sanitizer를 거치지 않는다.**
GitHub sanitizer가 제거하는 것은 README 마크다운 문서 안의 `<style>` 태그와 `style` 속성이다(6장 참고). 반면 `<img src="x.svg">`로 참조된 SVG는 README 문서의 일부가 아니라 **camo 프록시를 거쳐 로드되는 별도의 이미지 문서**다. 브라우저가 이 문서를 독립적으로 파싱하고 렌더링하므로, SVG 내부의 `<style>`은 소독 대상이 아니어서 그대로 동작한다. "README에서는 CSS를 못 쓴다"와 "SVG 카드 안에서는 CSS가 된다"가 공존하는 이유다.

**2. 미디어 쿼리는 브라우저가 그리는 순간 평가한다. (Chrome/Firefox/Edge 기준)**
SVG 안에 기본(라이트) 팔레트를 선언하고 `@media (prefers-color-scheme: dark) { ... }` 블록으로 다크 오버라이드를 함께 넣는다. 브라우저는 이미지를 그릴 때마다 이 조건을 평가해, 참이면 다크 규칙으로 덮어쓴다. 파일 내용은 항상 동일하므로 camo 캐시와 무관하다. 같은 캐시 파일이라도 기기 화면 모드를 토글하면 즉시 반대 팔레트로 다시 그려진다. 단 이것은 표준 동작을 하는 브라우저의 이야기이고, **Safari(WebKit)는 예외다** (아래 함정 2).

단, `<img>` 임베드 SVG는 "이미지 모드"라 제약이 있다. 스크립트 실행 금지, 외부 리소스 로드 금지(아바타를 base64로 내장하는 이유), 내부 링크 미동작(카드를 파일로 쪼개 `<a>`로 감싸는 이유). CSS와 미디어 쿼리는 렌더링의 일부라서 이 제약에 걸리지 않고 살아남는다.

**3. 투명 배경과 반투명 fill이 페이지에 녹아들게 한다.**
SVG 캔버스에서 도형을 그리지 않은 영역은 투명이라 GitHub 페이지 배경이 그대로 비친다. 카드 배경도 순수 배경색이 아니라 흰색을 라이트 55% / 다크 8% 불투명도로 깐다. 밑에 비치는 페이지 배경색과 섞이면서 라이트에서는 흰 카드, 다크에서는 은은한 유리 카드처럼 보인다. 팔레트 전환(2번)이 글자색을 바꾸고, 반투명(3번)이 배경을 페이지에 맞춰주는 분업이다.

**함정 1: 판정 기준은 GitHub 테마가 아니라 기기 화면 모드다.**
`prefers-color-scheme`은 방문자 **OS/브라우저의 다크 모드 설정**을 본다. 반면 GitHub 페이지의 배경은 **계정 테마 설정**(`html[data-color-mode]`)이 정한다. 진실의 원천이 두 개라서, 표준 동작 브라우저에서도 둘이 어긋나면(예: 시스템 라이트 + GitHub 테마 다크 고정) 다크 페이지 위에 라이트 팔레트 카드가 얹혀 일관되게 섞여 보인다.

**함정 2 (치명적): Safari는 `<img>` SVG의 미디어 쿼리를 지원하지 않는다.**
WebKit은 이미지로 임베드된 SVG에서 `prefers-color-scheme`을 지원하지 않는다(WebKit 버그 199134, 미해결). 지원되지 않는 기능이므로 평가 결과가 보장되지 않고, 실제로는 **비결정적**으로 동작한다. 2026-07 아이폰 Safari 실측: 다크 페이지에서 새로고침할 때마다 카드 4장이 서로 다른 팔레트 조합(예: 화이트/화이트/화이트/다크 -> 다크/화이트/다크/화이트)으로 무작위로 그려졌다. iOS는 모든 브라우저가 WebKit 엔진을 쓰므로 iOS 전체에 해당한다.

**투명도의 역할: 버그의 원인이 아니라 증폭기다.**
팔레트 선택 자체는 미디어 쿼리 문제지만, 투명 설계가 피해를 키운다. 라이트 팔레트(흰색 55% 불투명)가 다크 페이지 위에 잘못 그려지면 허옇게 뜬 카드가 되고, 다크 팔레트가 라이트 페이지에 그려지면 밝은 글자가 흰 배경에 묻힌다. 배경을 불투명으로 바꾸면 어느 팔레트로 그려지든 카드 자체는 항상 읽히지만(자기완결), 흰 카드와 검은 카드가 무작위로 섞이는 모습은 남는다. 즉 불투명화는 완화책이지 해결책이 아니다.

**구조적 해법은 조건 분기 제거다.**
GitHub 테마에 이미지를 묶는 공식 수단은 없고(`#gh-dark-mode-only` 프래그먼트 deprecated, `<picture>`도 OS 기준 + 어차피 SVG 내부가 아니라 파일 선택 단계라 함정 2와 별개), Safari에서는 미디어 쿼리 자체가 미지원이다. 모든 환경에서 동일하게 보이려면 미디어 쿼리를 빼고 **테마 중립 단일 팔레트**(양쪽 배경에서 읽히는 색)로 가는 것뿐이다. 조건이 없으면 잘못 평가될 것도 없다. `github-metrics.svg`가 실제로 이 방식이다. 테마 CSS 없이 고정 팔레트 + 투명 배경이라 적응은 못 하지만 어디서나(iOS 포함) 동일하게 보인다.

### 2단 레이아웃 (metrics 왼쪽 + 조직 섹션 오른쪽)

GitHub README는 CSS를 제거하므로 deprecated `align` 속성 플로트가 유일한 2단 배치 수단이다. 구조: `github-metrics.svg`와 카드 4장은 `align="left"` 플로트, **헤더(`org-header.svg`)는 플로트 없는 일반 인라인 이미지 + 뒤에 순수 `<br/>`**, 섹션 끝만 `<br clear="both">`로 해제한다(`clear`, `align`은 GitHub sanitizer 허용 속성).

헤더만 인라인인 이유: 플로트로 두면 중간 폭 격자에서 카드와 같은 행에 끼어 [헤더|카드] 행이 생긴다. 인라인 이미지는 (raw 텍스트와 달리) 짜부라지지 않고 통째로 다음 줄로 떨어지며, 뒤의 `<br/>`이 카드 시작점을 항상 헤더 다음 줄로 고정한다. `<br/>`은 clear가 없어 metrics 플로트에는 영향이 없다.

치수 (GitHub 프로필 README 컨테이너 실측 838px 기준):
- **GitHub이 `img[align=left]`에 `padding-right: 20px`를 주입한다** (DevTools 실측으로 발견). 플로트 박스 폭 = 이미지 폭 + 20. 이 20px가 열 사이 거터 역할을 하므로 SVG 내부 왼쪽 여백(PADL)은 0으로 없앴다.
- metrics: `width="420"` 속성으로 축소 표시 (자연 크기 480, 0.875배). 플로트 박스 440.
- 카드: 파일 폭 360, 높이 84 (본체 76 + 아래 여백 8). 플로트 박스 380.
- 헤더: 캔버스 폭 380 (내용 360 + 오른쪽 투명 여백 20). 인라인이라 GitHub 패딩 주입이 없으므로, 투명 여백으로 카드 플로트 박스(380)와 "metrics 옆에 들어가는 문턱"을 일치시킨다. 안 맞추면 800~820px 구간에서 헤더만 metrics 옆에 붙고 카드는 아래로 떨어져 분리된다 (실측으로 확인).
- 와이드 합계 440+380=820 <= 838 (여유 18px).
- **카드를 metrics보다 좁게 잡은 이유**: 2열 격자 구간의 폭 = metrics 박스 - 카드 박스. 절반 분할(392/392)로 하면 이 구간이 0이 되어 격자가 아예 안 뜬다. 420/360 분할로 760~820px 구간(60px)에서 격자가 산다.

폭별 동작 (838/790/400px 렌더링으로 검증, GitHub 주입 패딩 재현):
- **와이드(약 820px 이상)**: 헤더와 카드가 metrics 옆 오른쪽 열에 세로 스택.
- **중간(약 760~820px)**: metrics 아래에 헤더 행, 카드가 2행 2열 격자.
- **그 미만**: 헤더 행 + 카드 왼쪽 정렬 1열 세로 스택.

기각한 대안과 이유 (실측으로 확인):
- **raw 텍스트 제목/캡션이 플로트 옆으로 흐르는 안**: 몇 글자 폭만 남아도 그 틈에 흘러들어 세로로 짜부라진다. 그래서 제목/캡션을 `org-header.svg` 이미지로 만들었다.
- **카드를 오른쪽 플로트(`align="right"`)로 하는 안**: 중간 폭에서 카드가 아래로 떨어질 때 오른쪽 가장자리에 붙어 왼쪽에 큰 공백이 생기는 대각선 레이아웃이 된다.
- **헤더까지 왼쪽 플로트로 하는 안**: 중간 폭 격자에서 헤더가 카드와 같은 행에 낀다.
- **카드만 `<div>`로 감싸는 안**: 일반 div는 BFC가 아니라 플로트를 가두지 못해 레이아웃이 변하지 않는다.
- **metrics를 div로 감싸 아래 여백을 넣어 열 높이를 맞추는 안**: GitHub이 style을 제거해 div로는 여백을 못 만든다. `vspace`는 위아래 대칭이라 상단이 밀리고, `height` 속성 강제는 SVG가 세로 중앙 정렬돼 위에 공백이 생기고, 테이블 래퍼는 테두리가 생긴다. 대신 카드를 3줄로 압축해 열 높이를 줄이는 쪽으로 해결했다.

가로 간격은 CSS margin을 못 쓰므로 **카드/헤더 SVG 안에 투명 왼쪽 여백 12px(PADL)를 내장**해서 만든다 (파일 폭 342 = 여백 12 + 카드 330). 플로트가 맞붙는 두 지점(metrics-카드 사이, 격자 열 사이)이 이 여백으로 벌어지고, 세로 스택에서는 카드가 metrics의 내부 여백과 비슷하게 들어가 정렬이 자연스럽다.

왼쪽 플로트 안의 제약:
- **카드 사이 `<br clear="left">`를 쓸 수 없다.** metrics도 왼쪽 플로트라 같이 클리어돼 카드가 metrics 아래로 떨어진다. 대신 와이드에서는 "카드 옆 잔여 폭(35px) < 카드 폭"이라 자연히 세로로 쌓인다.
- **헤더와 카드의 높이를 동일(96px)하게 유지한다.** 행 높이가 어긋나면 격자가 지그재그가 된다. (헤더가 인라인이 되면서 격자에서는 빠졌지만, 와이드 오른쪽 열의 리듬을 위해 유지한다.)
- **불변식: metrics 표시 높이 > 마지막 카드의 시작 y(84x4=336px).** 이보다 짧아지면 마지막 카드가 "같은 높이의 더 왼쪽" 규칙에 따라 metrics 아래 x=0으로 떨어진다. metrics 표시 높이 = 자연 높이 x (420/480)이므로, **자연 높이가 약 384px 이상**이어야 한다 (half-year 기준 실측 440 -> 표시 385, 여유 약 49px). 열 전체(420px)가 metrics(385px)보다 길어지는 것 자체는 문제없다. 마지막 카드의 시작점만 넘기면 된다.

카드는 **480px 축소 표시가 아니라 396x76 네이티브 설계**다. width 속성 축소는 글자가 뭉개지기 때문이다. 폰트 크기는 유지하고, 설명은 한 줄로 유지하며, 리포 수는 이름 줄 오른쪽 끝(3줄 구성)에 배치해 카드 높이를 낮췄다. 열 높이를 줄여 위 불변식을 만족시키기 위한 구성이다.

### 조직 카드 자동 갱신 (org-cards.yml)

- `scripts/gen-org-cards.sh`가 `GET /orgs/{org}`에서 공개/비공개 리포 수를 읽어 카드 우하단에 "공개 N / 비공개 M"으로 표시한다.
- **비공개 수(`total_private_repos`)는 조직 소유자 권한 토큰으로 호출해야 응답에 포함된다.** CI에서는 `METRICS_TOKEN`을 `GH_TOKEN`으로 넘긴다. 토큰이 없으면 공개 수만 표시하는 fallback이 있다.
- 스케줄은 매일 00:30 UTC로, metrics(00:00 UTC)와 30분 오프셋을 둬 자동 커밋 push 충돌을 피한다.
- push 트리거는 `paths`로 생성기/워크플로 파일에 한정한다. 생성물(`org-*.svg`)은 paths에 없으므로 자기 커밋으로 재트리거되지 않는다.
- 로컬 실행: `GH_TOKEN=$(gh auth token) bash scripts/gen-org-cards.sh` (gh CLI 로그인 계정이 조직 소유자면 비공개 수까지 나온다.)
- 이 워크플로도 metrics처럼 `org-*.svg`를 자동 커밋하므로, 로컬과 origin이 발산하는 자동 커밋 소스가 하나 더 늘어난다. push 전 `git pull --rebase` 습관은 동일하게 적용한다.

### 기술 태그 (tag-*.svg, gen-tech-tags.sh)

README 맨 아래(`<br clear="both">` 뒤라 플로트 레이아웃과 간섭 없음)에 노션 태그 스타일의 기술 태그 필을 나열한다. 위 섹션과의 세로 간격은 CSS를 못 쓰므로 `<br clear="both">` 뒤에 `<br/>`를 붙여 빈 줄 하나로 만든다. 상세 설계와 결정 이력은 `docs/superpowers/specs/2026-07-15-tech-tags-design.md`에 있고, 여기는 운영에 필요한 핵심만 적는다.

- **스크립트의 `tags` 배열이 단일 소스다.** 태그 변경 = 배열 수정 + `bash scripts/gen-tech-tags.sh` 실행으로 끝난다. 스크립트가 `tag-*.svg` 생성, 배열에 없는 스테일 산출물 삭제, README의 `<!-- tech-tags:start/end -->` 마커 구간 재생성까지 한다. README 태그 목록을 손으로 편집하지 않는다.
- **CI 없음 (수동 실행).** org 카드와 달리 동적 데이터가 없어 재생성 트리거가 "배열 수정"뿐이고, 자동 커밋 소스를 늘리지 않기 위해서다.
- **팔레트는 org 카드와 동일한 테마 중립 모노톤.** 글자/아이콘 `#7d8590`, 필 `#8b949e` 7% 채움 + 40% 테두리. 미디어 쿼리를 쓰지 않는 이유는 위 "테마 대응 SVG" 절의 Safari 함정과 같다.
- **아이콘은 벡터 인라인.** `<img>` 임베드 SVG는 외부 리소스를 못 불러오므로 `icons/<slug>.svg` 캐시(1회 다운로드 후 커밋, 소스 핀 고정)에서 내부 요소를 추출해 태그 SVG에 내장한다. paint는 모노톤으로 정규화한다(`none` 보존, 다색/스타일 기반/외부 참조 아이콘은 검증 단계에서 거부).
- **아이콘 소스는 단색 변형만.** devicon `-plain` 계열(MIT, 고지: `icons/LICENSE-devicon.md`). 다색 아이콘을 한 색으로 칠하면 내부 문양이 사라지므로 다색 검출이 거부한다.
- **태그 간격은 SVG 내장 여백으로 만든다.** 캔버스 우측 2px + README 어절 공백 약 4px = 약 6px, 하단 6px가 줄바꿈 시 세로 간격. 인라인 이미지(align 없음)라 GitHub 패딩 주입이 없고 좁은 폭에서 자연 줄바꿈된다.

## 2. metrics 구성 개요 (metrics.yml)

- **`base: header`만 사용한다.** `activity`, `community`, `repositories` 섹션은 약점 지표(0 PR reviewed, 적은 stars/forks, sponsoring 0 등)를 노출하므로 제외한다. 강점은 아래 플러그인으로 따로 살린다.
- **plugins**
  - `plugin_isocalendar` (half-year): 입체 커밋 캘린더. 꾸준한 실행력을 약점 노출 없이 시각화한다.
  - `plugin_lines`: 코드 작성량. 구현 중심 역량 시그널.
  - `plugin_languages`: 사용 언어 분포. 직무 적합성(C# 최다)을 보여주는 핵심 자산이다.
- **트리거**: 매일 00:00 UTC 스케줄, `workflow_dispatch` 수동 실행, main push.
- **주의**: metrics 액션은 push마다 돌며 `github-metrics.svg`를 origin에 자동 커밋한다. 그래서 로컬과 origin이 자주 발산한다. 푸시 전에 `git pull --rebase origin main`으로 원격의 자동 커밋을 먼저 당긴다.

## 3. lowlighter/metrics 저장소 필터링 메커니즘

출처: `source/app/metrics/utils.mjs`의 `filters.repo()` 함수.
(원문: https://raw.githubusercontent.com/lowlighter/metrics/master/source/app/metrics/utils.mjs )

### 3.1 블랙리스트 (기본 동작, 정확 일치)

- `plugin_languages_skipped`는 전역 `repositories_skipped`를 상속한다.
- 기본 모드에서는 `owner/repo` 문자열이 **정확히 일치**하는 항목만 제외한다. 와일드카드 확장이 없다.
- 매칭 로직(요지): 패턴 배열에 `repo` 이름 또는 `owner/repo` 핸들이 들어 있으면 제외한다.

### 3.2 글로브 패턴 모드 (@use.patterns)

- skip 목록의 **첫 줄에 `@use.patterns`**를 넣으면 `minimatch` 기반 글로브 매칭이 켜진다.
- 이때 `siakun-testing/*` 같은 조직 단위 패턴이 동작한다.
- 매칭 로직(요지): `patterns[0] === "@use.patterns"`이면 각 항목을 `minimatch(handle, pattern)`으로 검사한다.

```yaml
plugin_languages_skipped: |-
  @use.patterns        # 반드시 첫 줄이어야 글로브가 켜진다
  siakun-private/*     # 조직 전체 제외
  bit-flo/*
```

### 3.3 화이트리스트는 네이티브로 없다

- languages 플러그인에는 "이 저장소만 포함"하는 include/allow-list 옵션이 **없다.** `plugin_languages_only` 같은 옵션은 존재하지 않는다.
- 특정 저장소를 강제로 **추가**하는 `plugin_languages_indepth_custom`은 있으나, 이건 분석 대상을 넓히는 용도지 좁히는 용도가 아니다. (skip을 무시하고 포함시킨다.)
- **화이트리스트 효과를 내는 방법**: 네이티브 화이트리스트가 없으므로, 조직 와일드카드로 "제외할 조직"을 지정해 사실상 화이트리스트처럼 운영한다.
  - 장점 1: 제외 조직에 새 스크래치 저장소가 생겨도 자동으로 빠진다. 목록을 계속 갱신할 필요가 없다.
  - 장점 2: `siakun-private`을 조직째 제외하면, 비공개 저장소 이름을 이 **공개** 설정 파일에 하나하나 노출하지 않아도 된다.

### 3.4 가시성(공개/비공개) 필터

- 와일드카드는 **이름 기반**이지 가시성 필터가 아니다. `siakun-private/*`는 그 조직을 통째로 빼는 것이지 "비공개라서" 빼는 게 아니다.
- 위치와 무관하게 "비공개 저장소만 전부" 제외하려면 skip 목록으로는 불가능하다. 가장 견고한 방법은 `METRICS_TOKEN`을 **공개 저장소 권한만 있는 토큰**으로 바꾸는 것이다. 그러면 metrics가 애초에 비공개 저장소를 못 보므로 통계가 자동으로 공개 기준이 된다. (토큰 교체는 저장소 Settings > Secrets에서 직접 한다.)

## 4. 현재 제외/포함 정책 (결정 사항)

언어 통계 기준이다. 커밋/기여 캘린더 같은 다른 지표에는 영향이 없다.

| 대상 | 처리 | 이유 |
|------|------|------|
| 모든 fork 저장소 | 제외 | `repositories_forks: no`. 남의 코드가 섞이지 않게 |
| `siakun-private/*` | 제외 | 비공개 조직. 리포 이름을 공개 설정에 나열하지 않도록 조직 단위로 제외 |
| `siakun-forks/*` | 제외 | 포크 모음 (forks: no로도 걸리지만 명시) |
| `bit-flo/*` | 제외 | 남의 조직/계정 협업 저장소 |
| `siakun/notedrop-share`, `siakun/LetterAI` | 제외 | 메인 계정의 데이터/AI 저장소 |
| `siakun-testing/*` | **포함** | 실험이지만 실제 작업물이라 통계에 넣는다 |
| `siakun-archive/*` | **포함** | 완료/보관한 실제 작업물이라 통계에 넣는다 |

집계 대상은 사실상 "메인 `siakun` 공개 저장소 + `siakun-testing` + `siakun-archive`"에서 위 개별 제외분을 뺀 것이다.

### 포함으로 생기는 부작용 (기억해 둘 것)

`siakun-testing`/`siakun-archive`를 포함하면 그 안의 데이터/노트북/학습 저장소도 같이 집계된다. 대표적으로:

- Jupyter Notebook / Python 비중을 올리는 저장소: `finetune`, `ImageColorAnalyzer`, `ArchiveChatAI`, `Prompts`, `Google-AI-Essentials`
- Java를 추가하는 저장소: `java_workspace`, `java-workspace`

C#을 앞세우려는 취지와 반대로 작동할 수 있다. 거슬리면 이 몇 개만 개별 제외(조직 와일드카드 + 소수 예외 하이브리드)한다. 또한 앞으로 `siakun-testing`에 순수 실험/데이터 저장소를 만들면 자동으로 집계에 들어오므로, 통계를 깔끔히 유지하려면 그런 건 개별 제외하거나 `siakun-private`로 옮긴다.

## 6. 함정과 주의점 (gotchas)

### GitHub README의 CSS 처리 (실측으로 확인한 지식)

- **인라인 CSS는 전부 제거된다.** `style` 속성, `<style>` 태그 모두 sanitizer가 지운다. 레이아웃 제어는 deprecated HTML 속성(`align`, `width`, `height`, `clear`, `vspace`, `hspace` 등 허용 목록)만 가능하다.
- **GitHub이 플로트 이미지에 여백을 주입한다.** 렌더러 자체 CSS로 다음 규칙이 적용된다.

  ```css
  .markdown-body img[align="left"]  { padding-right: 20px; }
  .markdown-body img[align="right"] { padding-left: 20px; }
  ```

  의도: 플로트 이미지는 글이 옆을 감싸 흐르므로, 텍스트가 이미지에 붙지 않게 감싸는 쪽에 숨통을 자동으로 넣는 배려다.
- **함정: 같은 폭의 이미지라도 플로트냐 인라인이냐에 따라 박스 폭이 20px 다르다.** `align` 속성 선택자에만 매칭되므로 인라인 이미지는 주입이 없다. 픽셀 단위로 폭 예산을 짜는 레이아웃(이 저장소의 2단 배치)에서는 이 20px 차이가 "옆에 들어가는 문턱"을 어긋나게 해 특정 폭 구간에서 요소들이 따로 논다. 예산 계산에는 항상 **플로트 박스 폭 = 이미지 폭 + 20**을 쓰고, 인라인 이미지는 투명 여백으로 문턱을 맞춘다.
- **검증 방법**: 로컬 미리보기(VS Code 등)에는 이 주입이 없어 GitHub과 다르게 렌더링된다. 목업으로 검증할 때는 위 규칙을 CSS로 재현해서 넣고, 최종 확인은 실제 GitHub에서 DevTools로 img 박스 폭을 실측한다. 이미지가 camo 프록시로 캐싱되므로 push 직후에는 강력 새로고침(Ctrl+Shift+R)이나 수 분 대기가 필요하다.

### metrics / 워크플로

- **블록 스칼라(`|-`) 안의 `#`는 주석이 아니다.** 리터럴 텍스트로 잡혀 패턴이 깨진다. skip 블록 안에는 패턴만 넣고, 설명은 블록 위 YAML 주석으로 둔다.
- **`@use.patterns`는 반드시 첫 줄이어야** 글로브 모드가 켜진다.
- **`plugin_languages_ignored`(언어 이름 지정 제외)는 신뢰할 수 없다.** `jupyter-notebook`을 넣어도 실제로는 제외되지 않아 5.59%로 계속 표시됐다. 그래서 이 옵션은 제거했다. 노트북을 빼려면 옵션이 아니라 해당 저장소를 skip하거나 category로 접근한다.
- **로컬에서 metrics SVG를 렌더할 수 없다.** `METRICS_TOKEN`이 필요하므로 CI에서만 생성된다. 결과 확인은 push하거나 Actions 탭에서 `workflow_dispatch`로 수동 실행한다.
